## Current Workspace Findings

### Critical

1. `[Critical] [workspace truth] [Affiliate + legal] [all breakpoints] The affiliate/legal workspace batch is partially implemented and currently breaks the repo baseline.`
   Evidence: `getReferralLinkAction()` now requires `agreedToTerms`, `AffiliatePortal` now links to `/affiliate-program`, and `AffiliateCTA` no longer uses `?auto_enroll=1`, but `app/(member)/account/affiliate/page.tsx` still passes `autoEnroll`. `npm run build` fails on that prop mismatch, and `npm run lint` fails in `components/affiliate/AffiliatePortal.tsx`.
   Action: finish the current affiliate/legal batch before treating any of those changes as current implementation truth.

2. `[Critical] [workspace truth] [Affiliate + legal] [all breakpoints] The planned public affiliate rules page is referenced in the UI but does not exist yet.`
   Evidence: the current workspace links to `/affiliate-program` from the affiliate CTA, the enrollment screen, and the earnings/payout area, but no such public route exists yet.
   Action: create the page before shipping the agreement flow.

### High

3. `[High] [workspace truth] [Affiliate UX] [all breakpoints] The affiliate portal has improved direction but remains in a volatile refactor state.`
   Evidence: `components/affiliate/AffiliatePortal.tsx` is still a large multi-behavior component, and the current workspace changes substantially rework `My Links`, enrollment, and `Share Kit` in one pass.
   Action: finish the current simplification batch, then split the portal before adding the next major affiliate architecture layer.

4. `[High] [workspace truth] [Affiliate + IA] [all breakpoints] Affiliate terms/agreement and affiliate architecture decisions are now partially encoded in code but not yet reflected in the operating docs.`
   Evidence: the workspace now assumes explicit agreement and a public rules page, while the prior truth docs still described a simpler member-only affiliate flow without that agreement layer.
   Action: refresh all truth docs before continuing implementation.

5. `[High] [decided next] [Affiliate architecture] [all breakpoints] The product is still member-first today, but the team is leaning toward a wider partner model.`
   Evidence: discussion has moved from “member affiliate portal only” to a likely Positives-hosted partner experience for non-members and spouses, while still using FirstPromoter underneath.
   Action: treat public partner support as a distinct architecture decision and design it intentionally rather than accreting it into the member-only portal.

### Medium

6. `[Medium] [live now] [Affiliate + reporting] [all breakpoints] FirstPromoter remains the source of truth, but link-management expectations have shifted several times and need one stable operating model.`
   Evidence: the recent sequence moved from local redirect links, to internal-only builder concepts, to a FirstPromoter-first link library, to a simplified member-facing link experience with optional source tags.
   Action: lock one canonical link-management model in docs before more portal work lands.

7. `[Medium] [live now] [Truth docs] [all breakpoints] The prior truth docs mixed shipped reality, local workspace changes, and future strategy.`
   Evidence: the older docs treated some affiliate behaviors as if they were current even after the workspace drifted into a new legal/onboarding direction.
   Action: use the new `current workspace / live now / decided next` model consistently across docs.

8. `[Medium] [decided next] [Legal + operations] [all breakpoints] Affiliate rules, prohibited behaviors, payout expectations, and enforcement rights need their own public surface.`
   Evidence: current public Terms and Privacy pages do not cover the affiliate-specific rules the team wants to rely on, including self-referrals, refund reversals, prohibited traffic, or termination rights.
   Action: add a dedicated affiliate program terms/info page and cross-link it from affiliate entry points.

9. `[Medium] [decided next] [Identity + lifecycle] [all breakpoints] The future partner/member reconciliation model is not yet defined in implementation terms.`
   Evidence: the team wants to support non-member affiliates later, and also wants same-email partner/member records to automatically connect.
   Action: document the future partner/member identity model explicitly before building it.
